2.  **Deconstruct "Speculative Decoding" (投机解码) in this Context:**
    - *What is speculative decoding?* In LLM inference, it's a technique where a small "draft" model generates multiple tokens quickly, and a large "target" model verifies them in parallel. If verified, they're accepted; otherwise, the sequence is truncated and the target model generates from the last accepted token.
    - *How does it apply to this system?* The system involves multiple LLM calls:
        - Coordinator LLM: Task decomposition (`squad_update`), progress reporting, nudge prompts.
        - Slave LLMs: Code development, `/loop` review, rebase handling, tool calling (`submit_to_squad`, `query_squad`).
    - Speculative decoding could be applied at the *inference layer* (speeding up individual LLM calls) or at the *system/agent layer* (speculative task execution, speculative tool calling, speculative DAG planning).
    - I need to analyze both angles but focus on what's most relevant to the PRD's architecture.

3.  **Analyze Pros & Cons (Brainstorming & Structuring):**

    *Pros (利):*
    - **Speed/Throughput:** Reduces latency for coordinator slave LLM calls. Faster task decomposition → faster scheduler ticks → faster overall project completion.
    - **Cost Efficiency:** Draft model is cheaper. Can handle routine slave tasks (simple code edits, standard rebase prompts) with a smaller model.
    - **Parallelism Alignment:** Matches the system's parallel slave architecture. Multiple slaves can run concurrently; faster per-task inference means higher effective concurrency.
    - **Speculative DAG Planning:** Coordinator could draft multiple DAG branches/speculations before committing to one, reducing back-and-forth with the user/LLM.
    - **Tool Call Optimization:** Speculative tool calling (drafting `submit_to_squad` or `/loop` calls before final verification) could streamline slave workflows.

    *Cons (弊):*
    - **Complexity vs. Reliability:** The PRD emphasizes deterministic state machines, ff-only merges, and strict validation. Speculative decoding introduces probabilistic acceptance/rejection, which could conflict with the "deterministic state machine" principle (公理 6: 类型消灭不可能态).
    - **State Machine Incompatibility:** `TaskStatus` transitions are strict. If a slave LLM speculatively calls `submit_to_squad` but the draft is rejected, the slave might be in an inconsistent state (e.g., thinks it submitted, but coordinator rejected). This violates the "command can be rejected, event cannot be denied" principle.
    - **Resource Contention:** Speculative decoding requires running two models (draft + target) or batching verification. In a resource-constrained environment (multiple slaves + coordinator), this could spike memory/CPU.
    - **Error Propagation:** If speculative tool calls (like `submit_to_squad`) are drafted and rejected, the slave might need to roll back state, complicating the clean `do-while ff` loop.
    - **Debuggability:** The PRD values traceability (NDJSON SSOT, git as second truth source). Speculative steps that don't commit to the event log create "ghost" states that are hard to reconcile during replay.
    - **Security/Safety:** Speculative tool calls could trigger side effects (e.g., `submit_to_squad` with wrong `commitSha`) if not carefully gated. The PRD's strict HTTP API with bearer tokens and serial queues assumes deterministic, verified calls.
