# MTP 投机解码消融实验与数学模型方案

## 1. 问题重述

之前的数学模型存在两个致命错误：
1. **基线数据错误**：摘要中的 $T_{\text{tgt}}(1) = 37.94$ ms 是解析 bug 造成的虚假数字。真实基线是 `server.log` 中的 **34.30 ms/token (29.15 t/s)**。
2. **CPU 核心数错误**：假设了 16 个有效计算核，实际只有 **8 个 P-Core**（0-15 是 8 物理核 × 2 超线程），HT 对计算密集的 MoE matmul 几乎无效。

## 2. CPU 拓扑真相

| 范围 | 类型 | 物理核数 | 超线程 |
|------|------|----------|--------|
| cpu 0-15 | P-Core | 8 | 每核 2 线程 |
| cpu 16-31 | E-Core | 16 | 每核 1 线程 |

`taskset -c 0-15` 绑的是 8 个 P-Core 的 16 个逻辑线程，但 MoE expert matmul 是计算密集型（AVX2），HT 兄弟线程争抢同一物理核的执行单元，加速 ≈ 0-15%。

## 3. 每 Token 物理成本分解

### 3.1 MoE 专家层（CPU 侧，40 层全部）

| 组件 | 每 token 读取量 |
|------|----------------|
| 8 个 expert × 3 矩阵 × 40 层 | 777.6 MiB |
| Shared expert × 40 层 | 97.2 MiB |
| Router × 40 层 | 16.2 MiB |
| **总 MoE 读取** | **891.0 MiB** |

实测带宽利用率：$891 \text{ MiB} / 34.3 \text{ ms} = 27.2 \text{ GB/s}$

### 3.2 Attention/GDN（GPU 侧）

| 组件 | 每 token 成本 |
|------|--------------|
| 10 层 GQA attention | ~0.3 ms（GPU 极快） |
| 30 层 GDN/SSM 状态更新 | GPU 内完成 |

### 3.3 瓶颈定位

实测 34.3 ms/token 中，MoE 内存读取理论下界为 15.9 ms（DDR4-3200 双通道 51.2 GB/s）。实际利用率 53%，**差距来自**：
- GPU↔CPU 同步开销
- mmap 页错误（`-ot exps=CPU` + mmap 的已知冲突）
- CPU compute（Q6_K dequant + matmul）
- 线程调度开销

### Phase 2：数学模型拟合

$$T_{\text{tgt}}(M, L_{\text{ctx}}) = \alpha + \beta \cdot M + \gamma \cdot M \cdot L_{\text{ctx}}$$

- $\alpha$：固定开销（GPU sync, kernel launch）
- $\beta$：每 token 边际成本（MoE compute + bandwidth）
- $\gamma$：每 token 每 context token 的 attention 扫描成本

投机加速比：

$$S(K) = \frac{T_{\text{tgt}}(1)}{\tau(K)}, \quad \tau(K) = \frac{K \cdot T_{\text{dft}} + T_{\text{tgt}}(K+1)}{E[N_{\text{acc}}(K)] + 1}$$

### 关键假设验证

如果 A4a（mmap）>> A1a（no-mmap），说明：
- 之前所有 benchmark 数据被 page fault 污染
- 真实 $T_{\text{tgt}}(M)$ 可能是报告值的 50-70%
- 投机解码在修正后可能实际有正收益
