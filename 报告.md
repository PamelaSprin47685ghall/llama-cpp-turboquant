# Ornith 35B TG 性能瓶颈消融实验与代码审计报告

## 0. 结论

**TG 性能瓶颈是 Q6_K 在 x86 上缺少 AVX2 interleaved repack kernel，导致 vec_dot 单行处理，AVX2 计算吞吐不足。**

实测 DDR5-4800 双通道（76.8 GB/s 峰值）仅利用 40%（~30 GB/s）。这不是内存带宽不足，而是 AVX2 dequant 计算吞吐不够快，无法充分利用带宽。

---

## 1. 真实消融实验数据

| 实验 | 配置 | TG ms/tok | TG t/s |
|------|------|-----------|--------|
| A3a | ctx=8K, t=8, no-mmap, graphs-OFF | 31.44 | 31.81 |
| A3b | ctx=8K, t=8, no-mmap, graphs-ON | 30.98 | 32.28 |
| A1a | ctx=262K, t=8, no-mmap, graphs-OFF | 33.76 | 29.62 |
| **A4a** | **ctx=262K, t=8, mmap, graphs-OFF** | **29.96** | **33.38** |
| A7a | ctx=262K, t=8, no-mmap, graphs-ON | 37.64 | 26.57 |

### 关键发现
- **mmap 在 262K ctx 反而更快**（29.96 vs 33.76 ms），与 8K 结论相反
- **CUDA graphs 在 262K ctx 有害**（37.64 vs 33.76 ms，慢 11.5%）
- **上下文长度对 TG 影响极小**（8K→262K 仅差 7.4%）

## 2. CPU 与内存拓扑

- **CPU**: 8 P-Core (cpu 0-15, 每核 2 HT) + 16 E-Core (cpu 16-31)
- **内存**: DDR5-4800 双通道 (2×32 GiB), 峰值 76.8 GB/s
- **L3**: 36 MiB

---

## 3. 物理 cost 模型

### 每 Token MoE 读取量
```
8 experts × (1024+512) × 2048 × 0.8125 bytes × 40 layers ≈ 788 MiB
```

### 带宽利用率
```
788 MiB / 29.96 ms = 27.6 GB/s = DDR5-4800 峰值的 36%
```

**这个利用率异常低**，暗示瓶颈不是内存带宽而是 AVX2 计算吞吐。

## 4. 代码审计根因

### 根因 #1：Q6_K 缺少 x86 AVX2 repack kernel（主要瓶颈）

`repack.cpp:4655` 中 `get_optimal_repack_type()` 对 Q6_K 的选择逻辑：

```cpp
} else if (cur->type == GGML_TYPE_Q6_K) {
    if (ggml_cpu_has_neon() && ggml_cpu_has_matmul_int8()) {
        return &q6_K_8x8_q8_K;  // ARM only!
    }
    if (ggml_cpu_has_neon() && ggml_cpu_has_dotprod()) {
        return &q6_K_8x4_q8_K;  // ARM only!
    }
    // x86: returns nullptr → falls to generic single-row path
}
```

对比 Q4_K（有 x86 优化）：
```cpp
} else if (cur->type == GGML_TYPE_Q4_K) {
    if (ggml_cpu_has_avx2()) {
        return &q4_K_8x8_q8_K;  // x86 AVX2 interleaved!
    }
}
```

**后果**：
- Q4_K 在 x86 上用 8 行 interleaved GEMM kernel，每次 activation load 产出 8 个输出元素
- Q6_K 在 x86 上用单行 generic `ggml_compute_forward_mul_mat_id`，每行独立重新加载 256 元素 Q8_K activation block
- `repack_q6_K_to_q6_K_8_bl()` 函数已存在（`repack.cpp:3420`），模板已实例化（`repack.cpp:4543`），但 x86 入口未连接

### 根因 #2：mul_mat_id 强制 nrc=1

`ggml-cpu.c:1502` 中 `ggml_compute_forward_mul_mat_id_one_chunk()` 始终以 `nrc=1` 调用 `vec_dot`，即使 `type_traits_cpu[Q6_K].nrows = 1`（x86），也无法利用多行批处理。普通 `mul_mat` 路径有 `num_rows_per_vec_dot` 优化，但 `mul_mat_id` 路径完全没有。

### 根因 #3：activation 量化线程分配不均（次要）

`ggml-cpu.c:1640-1655` 的 `#else` 分支按 ne10（K 维度）分配量化工作。对 down matmul（ne10=512, Q8_K block_size=256），只有 2 个线程得到工作，6/8 线程空闲。被禁用的 `#if 0` 分支按行分配，效果更好。

## 5. AVX2 计算吞吐分析

### Q6_K vec_dot 内核（`arch/x86/quants.c:2288`）

每个 Q6_K block（256 元素）的 AVX2 指令序列：

| 操作类别 | 指令数 | 关键端口 |
|----------|--------|----------|
| 内存加载 | 7 | Port 2/3 |
| 位提取（AND/SLLI/SRLI/OR） | 17 | Port 0/1/5 |
| maddubs（核心 dot product） | 4 | Port 0 (1/cycle) |
| scale shuffle | 4 | Port 5 (1/cycle) |
| cvtepi8_epi16 + madd_epi16 | 8 | Port 0+5 |
| add 累加 | 4 | Port 0/1 |
| **合计/block** | **~99** | |

关键瓶颈：Port 0 上有 ~30 个操作竞争，需要 ~30 cycles/block。
256 元素 / 30 cycles = **~8.5 elem/cycle/core**。

8 核 × 3.8 GHz × 8.5 = **259 Gelem/s** → 理论 ~7.7 ms/token。

但实测 30 ms，仍有 ~4× 差距。剩余差距来自：
- 专家权重非连续访问导致的 TLB miss 和 cache miss
- GPU↔CPU 图调度同步开销
- OpenMP 线程池 fork/join 开销

---

## 6. 对投机解码的影响

当前 T_tgt(1) ≈ 30 ms，T_tgt(M) ≈ M × 30 ms（线性）。
固定开销 α < 2%，无法被批量验证摊薄 → 投机解码数学上不可行。

如果添加 Q6_K x86 repack kernel（2-4× 加速）：
- T_tgt(1) 降至 ~10-15 ms
- 固定开销 α 占比提升至 ~5-10%
- 投机解码可能接近 break-even

---

## 7. 修复建议

| 优先级 | 修复 | 预期影响 |
|--------|------|----------|
| P0 | 在 `repack.cpp:4655` 添加 `ggml_cpu_has_avx2()` 分支，连接已有的 `q6_K_8x8_q8_K` repack trait | 2-4× MoE 加速 |
| P1 | 为 x86 编写 `block_q6_Kx8` 的 interleaved vec_dot kernel（类似 ARM 的 `ggml_gemm_q6_K_8x8_q8_K`） | 配合 P0 使用 |
| P2 | 修复 `ggml-cpu.c:1640` 量化线程分配（启用 `#if 0` 分支或改用行分配） | 1-3% |
| P3 | 在 `mul_mat_id_one_chunk` 中支持 `nrc>1` 多行批处理 | 5-10% |
