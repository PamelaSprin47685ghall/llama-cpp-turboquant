# Ornith 35B MTP 投机解码 26624 边界退化修复最终报告

---

## 1. 背景与问题描述

在 Ornith 35B（Qwen3.5-MoE 混合架构）推理与调优实践中，大上下文评测（Prefill 26401 tokens，生成 2048 tokens）时，生成至第 221 步（绝对位置 **`pos 26622`**，距离首个 256 对齐边界 26624 仅差 2 个 tokens）时，模型 logits 坍缩至单点分布，输出退化为无意义的 **`///` 重复循环**。该故障在关闭动态 KV 转码（DKVT）后消失，确认为 DKVT 的边界物理内存覆盖所致。

---

## 2. 根因剖析 (Root Cause Analysis)

物理显存覆写污染链条如下：

1. **逻辑与物理引用耦合悬空**：MTP 伴生草稿上下文（`layers.size() <= 1`，仅含 blk.40）为避免 draft updates 逻辑上反向越权清空或污染 parent context 序列位置，在其构造函数末尾执行 `if (is_mtp && other) other = nullptr;`。这导致 DKVT 转码初始化 `init_dkvt()` 阶段，因为 `other` 变 `nullptr`，伴生上下文无法借用主上下文的 union buffer 元数据，未能将 parent 状态（`is_transcoded_tg = true`）和物理大小继承。
2. **伴生激活区位置错误**：伴生上下文的 GPU 激活（`size_act_tg`）在共享 compute buffer `vram_union_block` 时，由于未设置 base offset，其 `base_offset` 默认为 0。这导致伴生上下文在执行 `llama_decode`（如 catch-up 评估）时，将激活张量分配在显存 `0` 偏移处，**直接覆盖了主上下文低地址处的 KV 缓存数据**（Layer 0, 1, 2... 的 V 缓存）。
3. **分配器重用（Graph Reuse）盲区**：`ggml_gallocr` 原生的 `ggml_gallocr_needs_realloc` 只根据节点数量与尺寸验证是否重分配。当 parent 发生 PP -> TG 转换且 `base_offset` 被设置时，分配器因拓扑未变直接复用先前 PP 阶段缓存在 0 偏移处的物理分配，使得 activations 覆写主上下文 KV Cache 的故障静默发生，不报 OOM / CUDA error 仅产生数值污染。

---

## 3. 修复方案与代码实现

### 3.1 引入正交物理指针解耦
在 `src/llama-kv-cache.h` 增加 `dkvt_parent` 指针并在 `llama-kv-cache.cpp` 构造列表中执行 `dkvt_parent(other)` 初始化。即使 logical `other` 在 MTP 分支下被清空，物理引用的 parent context 仍能被 DKVT 完美继承。

### 3.2 伴生上下文 base_offset 借用补全
在 `src/llama-kv-cache-dkvt.cpp` 中将所有 DKVT 相关的 `other` 引用重构为 `dkvt_parent`。当 `layers.size() <= 1` 触发早期返回时，如果 `dkvt_parent` 存在且活跃，强制同步 `vram_union_block` 及 TG 阶段的 KV 规格，并调用 `dkvt_apply_union_compute_cap(sched)` 将其 scheduler 激活偏移安全推至 `base_offset`（即 `dkvt_v_size_tg + dkvt_k_size_tg`）以外，杜绝物理越界。

### 3.3 强行重置分配器缓存
在 `ggml/src/ggml-alloc.c` 的 `ggml_gallocr_set_borrowed_compute_base_offset` 中增加平移校验：
```c
void ggml_gallocr_set_borrowed_compute_base_offset(ggml_gallocr_t galloc, int buffer_id, size_t base_offset) {
    GGML_ASSERT(buffer_id >= 0 && buffer_id < galloc->n_buffers);
    if (galloc->buf_tallocs[buffer_id] != NULL) {
        if (galloc->buf_tallocs[buffer_id]->base_offset != base_offset) {
            ggml_dyn_tallocr_set_base_offset(galloc->buf_tallocs[buffer_id], base_offset);
            // 强行重置分配节点计数，迫使 needs_realloc 返回 true 触发重新编排
            galloc->n_nodes = 0;
            galloc->n_leafs = 0;
        }
    }
}
```

## 4. TDD 验证机制与测试结果

我们新建了回归测试文件 **`tests/test-dkvt-companion-offset.cpp`** 并注册至 `CMakeLists.txt`。

- **测试设计**：
  1. 构造一个包含 4 层的 `parent_cache`，并填充 mock GPU 显存 buffer；
  2. 模拟 parent 转码至 TG 状态，计算其 TG 规格下的 `expected_base_offset = 1049664` 与 `expected_cap = 3146816`；
  3. 构造伴生上下文 `companion_cache`（强制层数 `layers.size() = 1`），使用 CPU 线程调度器绑定物理共享显存块；
  4. 调用 `companion_cache.init_dkvt()`。
- **校验点**：
  - 断言 `companion_cache.is_transcoded_tg == true`（同步正确）；
  - 断言 `sched->galloc->buf_tallocs[0]->base_offset == 1049664`（物理偏移正确平移）；
  - 断言 `sched->galloc->buf_tallocs[0]->cap == 3146816`（能力上限制动正确）。
