## 3. 已完成的核心架构与 Bug 修复

### 3.1 伴生草稿上下文（MTP）NaN 概率修复
* **病因**：MTP 伴生上下文（layers=1, blk.40）由于 `is_mtp` 开启，在构造函数中其 `other` 指针被强制置为 `nullptr`，导致无法调用 `init_dkvt_borrow()`。这使得伴生上下文不会运行自己的转码内核，但 `dkvt_bind_tg()` 仍然被触发并将它的张量类型改为了 `F16/Q8_0`，而内存数据仍然是 PP 态的 `turbo` 格式，从而产生数据格式错位，导致第一步生成起草即输出 `nan`。
* **修复**：在 `llama_kv_cache::init_dkvt` 的第一行添加短路逻辑，**当 `layers.size() <= 1` 时直接禁用 DKVT**。因为伴生层体积极小（~18MB），不参与动态转码能彻底避免多图 layout 同步的冲突，概率直接恢复正常。

### 3.2 解决 GPU 转码原位重叠冲突（镜像布局 Mirror Layout）
* **病因**：在 in-place 转码（PP KV -> TG KV）中，由于 TG 的 `F16` 格式比 PP 的 `turbo` 格式大，TG K 的写入会与未读取的 PP K 数据发生 569 MiB 的大范围物理重叠。原版代码采用 `layers.size() - 1 - idx` 逆序遍历层无法完全消除跨层重叠导致的源数据损坏。
* **修复**：
  - 重新实现了 **镜像布局（Mirror Layout）**：在 TG 状态下强制绑定 KV 位于 `[0, kv_total_tg)`，计算图分配在 `[kv_total_tg, kv_total_tg + size_act_tg)`。
  - 在分配器中实现了 `base_offset`（`ggml-alloc.c` / `ggml-backend.cpp`）：告诉图分配器在 TG 阶段避开底部的 KV 显存，从 `base_offset` 往上分配计算张量。
  - **并发转码优化**：直接调用成熟的 CUDA 全层转码 API `ggml_cuda_transcode_k/v`，一次性发射所有层的异步 GPU 内核，移除了 per-cell 极其低效的行同步（`cudaStreamSynchronize`），将转码阶段耗时从 5.3s 缩短至毫秒级。

### 3.3 投机解码 recurrent 状态回滚 Bug 修复
* **病因 1 (n_rs_seq 限幅)**：投机长度 $n_{\text{max}} = 3$ 时验证批次为 4 字符。如果全部被拒绝，target model 必须回滚 4 步。但 `n_rs_seq` 默认被设置为 `3`（`draft.n_max`），导致 4 步回滚直接返回 `false` 失败。
  - *修复*：修改 `common/common.h` 的 `need_n_rs_seq()` 返回 `draft.n_max + 1` (4)。
* **病因 2 (s_copy 读空)**：在 rollback 发生时，`seq_rm` 会将 cells 的 `seq_id` 清除以释放位置。但 `s_copy` 在 graph build 期间从 `cells[cell_idx].seq_id` 获取序列 ID；如果被清除为空，它将完全忽略 rollback 索引 `rs_idx` 并回退到 slot 0，导致回滚失效。
  - *修复*：修改 `llama-memory-recurrent.cpp` 中的 `s_copy`，改从当前执行的 `ubatches[i_next]` 批次中获取真实的序列 ID，完美绕过了 empty cells 的限制。
* **病因 3 (rs_idx 提前清空)**：`rs_idx[seq]` 原本在 `s_copy` 内部被重置为 `0`。但这导致在 40 层的模型构建中，第一层第一算子（layer 0 conv）读完后，剩余 39 层的所有 attn 和 conv 层都读到了 `0`，使得只有一层回滚成功。
  - *修复*：在 `llama_memory_recurrent_context` 析构函数中执行 `rs_idx` 清空，确保整个 Graph 均读取到相同的回滚偏移。
* **病因 4 (Checkpoint 强制)**：为了在 GPU 上提供 100% 绝对干净的循环状态回滚（避免 shift register 的 slots 污染），修改 `tools/server/server-context.cpp`，**强制 target context 采用 checkpoint 回滚** (`use_ckpt_tgt = true`)，仅需 0.1ms 的 GPU copy，杜绝了 ssm-states 在 long loop 下的偏移污染。

## 4. 遗留问题与未完成工作（移交下一个 Agent）

### 4.1 长文本退化 (At Pos 26622 -> ///)
在进行大上下文 PRD 测试（prefill 26401 tokens，生成 2048 tokens）时，虽然 prefill 速度（`862.89 t/s`）和首包延迟（`30.596 s`）表现优异，但在 TG 解码生成了约 221 个 token 到达 **`pos 26622`** 时，大模型的输出开始陷入退化的 **`///` 重复循环**。

#### 诊断进展与排查方向：
1. **`pos 26622` 的特殊物理意义**：
   - 26,401（预填充） padded to 256 = **`26,624`**。
   - `pos 26622` 距离第一个 256 单元对齐边界（`26,624`）仅差 2 个 tokens！
   - 当生成到 `26622` 时，MTP 伴随模型需要起草 `26622, 26623, 26624`，这意味着在这一步中，KV Cache 正好需要**向外分配一个新的 256-cell 的物理块**（跨越 26,624 边界）。
2. **可能的内存越界或越位 (OOB) 损坏**：
   - 虽然 `size_act_pp` 和 `size_act_tg` 的内存容量被镜像布局完美隔开，但当 KV Cache 触发向外申请新的 256 块（Block 104）时，分配器 `find_slot` 或 `set_rows` 在 GPU 侧写入的物理地址可能由于 TG 偏移的重计算或 `dkvt_bind_tg` 存在某些计算微偏，**踩到了临近的计算图激活内存**，从而引发了静默的数据污染。
3. **SSM 循环层状态可能存在潜在的 GPU-CPU 同步丢失**：
   - 主模型的 SSM 状态是在 GPU 上更新的，而 speculative 逻辑会调用 rollback。需在 `delta-net-base.cpp` 的 shift 复制中做更深入的断点测试，确认在 Mirror 状态下，GPU 状态在长循环时是否还存留有其他偏置。
