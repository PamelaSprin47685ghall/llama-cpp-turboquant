# 动态 KV 转码（DKVT）双向对撞式显存别名实现计划

---

## 1. 物理账本与首尾对撞布局

针对 Qwen 3.5 MoE（总层数 40 层，其中 10 层 GQA，30 层 GDN），在关闭 Xorg 等图形界面（无损释放约 0.8GB，无头驱动/系统保留仅需 0.15GB）的 8GB 显存单卡上，上下文写死 256000 且物理批次 3072 下，我们采用双向栈式显存复用（Bidirectional Memory Aliasing）设计。

静态分配唯一的显存 Flat Buffer 空间：
- Union Buffer 物理总大小 = max(PP 阶段需求, TG 阶段需求) = max(2.7G 计算图 + 0.915G turbo KV, 0.1G 计算图 + 3.74G f16q8_0 KV) = 3.84 GB。

在推理执行流中，计算图从 Union Buffer 的头部向后生长（正向使用），KV Cache 从 Union Buffer 的尾部向前生长（逆向使用），两者在空间上“对撞”但绝不重叠相遇。

```
                    vram_union_block (静态分配 3.84 GB)
               ptr_start = 0                              ptr_end = 3.84 GB
               │                                                          │
               ▼                                                          ▼
PP 阶段布局    ├───────────────┬──────────────────────────┬───────────────┤
               │ 计算图 (2.7G) │    安全空隙 (0.225G)     │ turbo KV (0.915G) 
               │ (正向使用)    │                          │ (逆向使用)    │
               ├───────────────┼──────────────────────────┴───────────────┤
TG 阶段布局    │ 计算图 (0.1G) │     K: f16 + V: q8_0 混合 KV (3.74 GB)   │
               │ (正向使用)    │     (逆向使用)                           │
               └───────────────┴──────────────────────────────────────────┘
```

---

## 2. 物理空间无交叉判定

1. PP 阶段无交叉判定：
   - 正向计算图边界：[0.00 GB, 2.70 GB]
   - 逆向 turbo4/turbo2 KV 边界：[2.925 GB, 3.84 GB]
   - 空间物理距离：2.925G - 2.7G = 0.225 GB (230 MiB 绝对空闲区)，安全无交叉。
2. TG 阶段无交叉判定：
   - 正向计算图边界：[0.00 GB, 0.10 GB]
   - 逆向 f16/q8_0 KV 边界：[0.10 GB, 3.84 GB]
   - 空间物理距离：0.1G - 0.1G = 0.00 GB (物理恰好对齐接轨，不重叠)，安全无交叉。

## 3. 四步走落地实施计划

### 阶段一：静态Flat Buffer首尾绑定
1. 在初始化阶段，绕开分配器静态申请一个 3.84 GB 的物理显存 Flat Buffer，命名为 vram_union_block，首物理地址为 ptr_start，尾物理地址为 ptr_end = ptr_start + 3.84G。
2. 扫描并定位 10 层 GQA 的 Key/Value Tensors，为这些 Tensor 的 flags 属性添加 GGML_TENSOR_FLAG_EXT 标志。
3. **首尾数据指针固定分配**：
   - 将 10 层 GQA Tensors 的 data 指针初始且永久绑定在逆向生长对应的物理地址偏置上（model.layers[l].k->data = ptr_end - offset_k[l]；model.layers[l].v->data = ptr_end - offset_v[l]）。
   - 指针在推理全生命周期内固定不变，省去了 PP -> TG 切换时改写指针的逻辑开销。

### 阶段二：PP阶段低比特写入
1. 启动大 batch（3072）Prefill 阶段前向计算。
2. ggml-alloc 分配器将计算图的工作缓冲区指向 vram_union_block 头部 ptr_start，正向使用前 2.7 GB 空间。
3. 异构计算流水线在生成 Key/Value 时，直接将数据以 turbo4 和 turbo2 格式，从 ptr_end 尾部逆向写入到 [2.925 GB, 3.84 GB] 的显存物理区间内。

### 阶段三：临界转码（PP -> TG 边界）
1. 当大 batch prefill 结束后，解构并释放正向 PP 计算图对 vram_union_block 的占用。
2. 启动一次性动态转码 CUDA Kernel（transcode_turbo_to_mixed_kernel）。该 Kernel 运行在 GPU 内部，读取尾部 [2.925 GB, 3.84 GB] 的 turbo 格式源数据，在显存内部直接进行反量化与重量化，原位向左膨胀生成 2.44 GB 的 f16 格式 Key 与 1.30 GB 的 q8_0 格式 Value，直接覆盖写入到逆向区间 [0.10 GB, 3.84 GB] 内。
3. 由于读写指针都在显存内部且符合逆向重合，转码耗时仅为 23.2 ms（完全在显存内部 272 GB/s 带宽下进行，无任何 PCIe 总线参与，用户完全无感）。
4. 将 TG 阶段分配器（tg_gallocr）的 buffer 地址指向 vram_union_block 的头部 ptr_start，正向使用前 0.1 GB 空间。

### 阶段四：TG阶段高带宽无阻碍生成
1. 启动自回归生成阶段（batch=1），开启自推测解码。
2. 由于 KV Cache 从第一天起就常驻于 vram_union_block 尾部，注意力算子将直接以 272 GB/s 满显存带宽在本地执行，TG 吞吐量将达到 50+ t/s 的极限水平。
3. 伴随每个 Token 的自回归产出，新生成的 KV 缓存直接以 K: f16, V: q8_0 格式逆向追加写入到 vram_union_block 尾部。

## 4. 刚性约束与错误防御
1. 并发独占安全：此方案仅支持单会话（Single-session）独占模式。多 Slot 并发时，PP 阶段的计算图会与 TG 阶段的 KV 显存区间发生重叠撕裂，必须在 llama-server 初始化时强行限制 n_parallel = 1。
2. 共享图伴生校验：对于 MTP 计算图共享补丁，伴生预测上下文的调度器（share_compute_buffers_with）在初始化及运行时，必须在 has_layer 哨兵中增加对 GGML_TENSOR_FLAG_EXT 外部指针的校验，跳过对 borrowed buffers 的物理重分配，杜绝 Use-After-Free 崩溃。
