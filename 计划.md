# MTP speculative decoding regression and gibberish output fix plan

## 1. Issue Description

After the DKVT (Dynamic KV Transcode) implementation, MTP speculative decoding started normally but quickly decayed into nonsense (garbage predictions) towards the end of long generations, leading to a significant drop in draft acceptance rates and text generation (TG) speed.

---

## 2. Root Cause Analysis

We identified two major issues causing this state corruption:
1. **Target context nextn embeddings reordering swap bug:**
   When `embeddings_nextn_masked = false` (unmasked mode, used for target context `ctx_tgt` under MTP), `embd_nextn` size is `n_batch * n_embd` and stores nextn embeddings densely in batch chronological order. However, `llama_context::output_reorder()` still performed swaps on the first `n_outputs` rows of `embd_nextn` using `output_swaps`. This corrupted the order of nextn embeddings, feeding wrong hidden states to MTP and causing it to predict nonsense.
2. **Draft context KV cache duplication bug:**
   During the MTP speculative verification step, `ctx_dft` catch-up decode (`llama_decode`) was called in `process()` without removing draft-phase cells first. This caused `find_slot()` to allocate new empty cells elsewhere for the same positions, resulting in duplicate KV Cache entries (one set from `draft()`, one set from `process()`). This duplication corrupted the self-attention calculation of the draft model.

---

## 3. Implemented Fixes

1. **Unmasked nextn embeddings reordering bypass:**
   Modified `src/llama-context.cpp` `output_reorder()` to only apply `output_swaps` to `embd_nextn` when `cparams.embeddings_nextn_masked == true` (masked mode).
2. **Draft context KV Cache residue cleanup:**
   Modified `common/speculative.cpp` MTP `process()` to call `llama_memory_seq_rm` and clear draft-phase cells in `ctx_dft` starting from the first token's position before the catch-up decode.
3. **MTP draft token printing:**
   Added unconditional token printing to `stderr` in `common/speculative.cpp` `draft()` to log each drafted token and its probability.
4. **Draft KV Cache CPU offloading:**
   Modified MTP companion context initialization in `tools/server/server-context.cpp` to correctly copy `n_gpu_layers` to `cparams_mtp` and disable `offload_kqv` when `n_gpu_layers == 0`, preventing CUDA VRAM OOM on 8GB GPU.

---

## 4. First-Principles Non-linear Latency Model in TG Phase for $(K, p_{\text{min}}, n_{\text{min}})$

Under `-ot "exps=CPU"`, target verification latency $T_{\text{tgt}}(M)$ exhibits strong non-linear and non-monotonic hardware scheduling behavior due to CPU OpenMP thread count limits (16 threads for 8 experts per token = 32 expert computations at $M=4$). 
Measured TG latencies (excluding warmup):
* $T_{\text{tgt}}(1) = 37.94$ ms (baseline)
* $T_{\text{tgt}}(2) = 72.84$ ms ($K=1$)
* $T_{\text{tgt}}(3) = 103.81$ ms ($K=2$, high latency due to single-threaded CPU bottleneck)
* $T_{\text{tgt}}(4) = 81.35$ ms ($K=3$, low latency due to full OpenMP parallelization)
* $T_{\text{tgt}}(6) = 158.80$ ms ($K=5$)

Minimizing expected latency per token:
$$\tau(K, p_{\text{min}}) = \frac{K \cdot T_{\text{dft}} + T_{\text{tgt}}(K + 1)}{E[N_{\text{acc}}(K)] + 1}$$
with $T_{\text{dft}} = 1.50$ ms and empirical accepted tokens $E[N_{\text{acc}}(K)]$:

* **For $K=3$ (`n_max = 3`):** Speedup is **1.054x** (only config with speedup), and Token Acceptance Rate is **46.17%** (to reach >50%, we must set `--spec-draft-p-min 0.20`, raising it to **52.3%**).
* **For $K=2$ (`n_max = 2`):** Speedup is **0.7854x** (slowdown due to $T_{\text{tgt}}(3) = 103.81$ ms bottleneck), and Token Acceptance Rate is **60.55%**.
* **For $n_{\text{min}}$ (`n_min`):** Since $r = T_{\text{dft}}/T_{\text{tgt}}(1) \approx 0.040$, any step $k \ge 1$ with $>4.0\%$ acceptance probability yields positive speedup. Thus, **`n_min = 0` (or 1)** is mathematically optimal.
* **For $p_{\text{min}}$ (`p_min`):** The marginal cost model dictates starting step $i$ iff $p_i > \frac{r \cdot S(K)}{\gamma \cdot P(i-1)}$, which gives $p_{\text{min}} \approx 0.05$.

Thus, the optimal server configuration is:
* **`--spec-draft-n-max 3`**
* **`--spec-draft-p-min 0.20`**
* **`--spec-draft-n-min 0`**
