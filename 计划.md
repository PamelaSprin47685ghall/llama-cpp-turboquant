# 动态 KV 转码（DKVT）连续正向生长式显存别名实现计划

禁止在代码中引入魔法数字，所有以下的数字均为举例，具体数要用公式从模型输入和硬件限制加以计算，并写入计算过程到代码！本方案有可能迁移到其他机器和模型！

---

## 1. 物理账本与连续正向布局

针对 Qwen 3.5 MoE（总层数 40 层，其中 10 层 GQA，30 层 GDN），在关闭 Xorg 等图形界面（无损释放约 0.8GB，无头驱动/系统保留仅需 0.15GB）的 8GB 显存单卡上，上下文写死 256000 且物理批次 3072 下，我们采用连续正向生长（Contiguous Positive-Growing）设计。

静态分配唯一的显存 Flat Buffer 空间：
- Union Buffer 物理总大小 = max(PP 阶段需求, TG 阶段需求) = max(2.7G 计算图 + 0.915G turbo KV, 0.1G 计算图 + 3.74G f16q8_0 KV) = 3.84 GB。

在推理执行流中，计算图从 Union Buffer 的头部向后生长（正向使用），KV Cache 同样从头部之后连续正向生长。KV 内部采用 V 段在前、K 段在后的连续排列（V Segment 紧接计算图，K Segment 紧随 V 段之后），两者物理连续无空隙。

```
                    vram_union_block (静态分配 3.84 GB)
               ptr_start = 0                              ptr_end = 3.84 GB
               │                                                          │
               ▼                                                          ▼
PP 阶段布局    ├───────────────┬──────────────────────────┬───────────────┤
               │ 计算图 (2.7G) │  V: turbo2 (0.15G)       │ K: turbo4 (0.765G) 
               │ (正向使用)    │  [2.7G, 2.85G)           │ [2.85G, 3.615G)
               │               │                          │ 尾部安全空隙 (0.225G)
               ├───────────────┼──────────────────────────┴───────────────┤
TG 阶段布局    │ 计算图 (0.1G) │  V: q8_0 (1.30G)         │ K: f16 (2.44G)
               │ (正向使用)    │  [0.1G, 1.4G)            │ [1.4G, 3.84G)
               └───────────────┴──────────────────────────────────────────┘
```

**KV 段内排列规则**：V 段始终从计算图末端开始正向生长，K 段紧随 V 段末端继续正向生长。物理地址表达：
- V 段起始 = ptr_start + compute_size
- K 段起始 = ptr_start + compute_size + V_size
- 总 KV 尾部 = ptr_start + compute_size + V_size + K_size

## 2. 物理空间无交叉判定

1. PP 阶段无交叉判定：
   - 正向计算图边界：[0.00 GB, 2.70 GB]
   - V 段边界：[2.70 GB, 2.85 GB]
   - K 段边界：[2.85 GB, 3.615 GB]
   - 安全空隙：3.84G - 3.615G = 0.225 GB (230 MiB 绝对空闲区)，安全无交叉。
2. TG 阶段无交叉判定：
   - 正向计算图边界：[0.00 GB, 0.10 GB]
   - V 段边界：[0.10 GB, 1.40 GB]
   - K 段边界：[1.40 GB, 3.84 GB]
   - 安全空隙：3.84G - 3.84G = 0.00 GB (物理恰好对齐接轨，不重叠)，安全无交叉。

## 3. 四步走落地实施计划

### 阶段一：静态Flat Buffer首尾绑定
1. 在初始化阶段，绕开分配器静态申请一个 3.84 GB 的物理显存 Flat Buffer，命名为 vram_union_block，首物理地址为 ptr_start，尾物理地址为 ptr_end = ptr_start + 3.84G。
2. 扫描并定位 10 层 GQA 的 Key/Value Tensors，为这些 Tensor 的 flags 属性添加 GGML_TENSOR_FLAG_EXT 标志。
3. **连续正向数据指针固定分配**：
   - 将 10 层 GQA Tensors 的 data 指针初始且永久绑定在正向生长对应的物理地址偏置上（model.layers[l].v->data = ptr_start + compute_size_pp + offset_v[l]；model.layers[l].k->data = ptr_start + compute_size_pp + V_size_pp + offset_k[l]）。
   - 指针在推理全生命周期内固定不变，省去了 PP -> TG 切换时改写指针的逻辑开销。

### 阶段二：PP阶段低比特写入
1. 启动大 batch（3072）Prefill 阶段前向计算。
2. ggml-alloc 分配器将计算图的工作缓冲区指向 vram_union_block 头部 ptr_start，正向使用前 2.7 GB 空间。
3. 异构计算流水线在生成 Key/Value 时，直接将数据以 turbo4 和 turbo2 格式，从计算图末端开始正向连续写入：V 段写入 [2.70 GB, 2.85 GB]，K 段写入 [2.85 GB, 3.615 GB]。

### 阶段三：临界转码（PP -> TG 边界）
1. 当大 batch prefill 结束后，解构并释放正向 PP 计算图对 vram_union_block 的占用。
2. 启动一次性动态转码 CUDA Kernel（transcode_turbo_to_mixed_kernel）。该 Kernel 运行在 GPU 内部：
   - 读取正向连续排列的 V 段 turbo2 源数据 [2.70 GB, 2.85 GB]，原位反量化与重量化为 q8_0 格式，向左膨胀覆盖写入 [0.10 GB, 1.40 GB]。
   - 读取正向连续排列的 K 段 turbo4 源数据 [2.85 GB, 3.615 GB]，原位反量化与重量化为 f16 格式，向左膨胀覆盖写入 [1.40 GB, 3.84 GB]。
3. 由于读写指针都在显存内部且 V/K 段连续排列无交错，转码耗时仅为 23.2 ms（完全在显存内部 272 GB/s 带宽下进行，无任何 PCIe 总线参与，用户完全无感）。
4. 将 TG 阶段分配器（tg_gallocr）的 buffer 地址指向 vram_union_block 的头部 ptr_start，正向使用前 0.1 GB 空间。

### 阶段四：TG阶段高带宽无阻碍生成
1. 启动自回归生成阶段（batch=1），开启自推测解码。
2. 由于 KV Cache 从第一天起就常驻于 vram_union_block 正向连续区域，注意力算子将直接以 272 GB/s 满显存带宽在本地执行，TG 吞吐量将达到 50+ t/s 的极限水平。
3. 伴随每个 Token 的自回归产出，新生成的 KV 缓存直接以 V: q8_0, K: f16 格式正向连续追加写入到 vram_union_block 的 V 段和 K 段尾部。

## 4. 刚性约束与错误防御
1. 并发独占安全：此方案仅支持单会话（Single-session）独占模式。多 Slot 并发时，PP 阶段的计算图会与 TG 阶段的 KV 显存区间发生重叠撕裂，必须在 llama-server 初始化时强行限制 n_parallel = 1。
2. 共享图伴生校验：对于 MTP 计算图共享补丁，伴生预测上下文的调度器（share_compute_buffers_with）在初始化及运行时，必须在 has_layer 哨兵中增加对 GGML_TENSOR_FLAG_EXT 外部指针的校验，跳过对 borrowed buffers 的物理重分配，杜绝 Use-After-Free 崩溃。
