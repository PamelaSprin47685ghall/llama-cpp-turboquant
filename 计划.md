# MTP speculative decoding regression and gibberish output fix plan

## 1. Issue Description

After the DKVT (Dynamic KV Transcode) implementation, MTP speculative decoding started normally but quickly decayed into nonsense (garbage predictions) towards the end of long generations, leading to a significant drop in draft acceptance rates and text generation (TG) speed.

---

## 2. Root Cause Analysis

We identified two major issues causing this state corruption:
1. **Target context nextn embeddings reordering swap bug:**
   When `embeddings_nextn_masked = false` (unmasked mode, used for target context `ctx_tgt` under MTP), `embd_nextn` size is `n_batch * n_embd` and stores nextn embeddings densely in batch chronological order. However, `llama_context::output_reorder()` still performed swaps on the first `n_outputs` rows of `embd_nextn` using `output_swaps`. This corrupted the order of nextn embeddings, feeding wrong hidden states to MTP and causing it to predict nonsense.
2. **Draft context KV cache duplication bug:**
   During the MTP speculative verification step, `ctx_dft` catch-up decode (`llama_decode`) was called in `process()` without removing draft-phase cells first. This caused `find_slot()` to allocate new empty cells elsewhere for the same positions, resulting in duplicate KV Cache entries (one set from `draft()`, one set from `process()`). This duplication corrupted the self-attention calculation of the draft model.

---

## 3. Implemented Fixes

1. **Unmasked nextn embeddings reordering bypass:**
   Modified `src/llama-context.cpp` `output_reorder()` to only apply `output_swaps` to `embd_nextn` when `cparams.embeddings_nextn_masked == true` (masked mode).
2. **Draft context KV Cache residue cleanup:**
   Modified `common/speculative.cpp` MTP `process()` to call `llama_memory_seq_rm` and clear draft-phase cells in `ctx_dft` starting from the first token's position before the catch-up decode.
3. **MTP draft token printing:**
   Added unconditional token printing to `stderr` in `common/speculative.cpp` `draft()` to log each drafted token and its probability.
4. **Draft KV Cache CPU offloading:**
   Modified MTP companion context initialization in `tools/server/server-context.cpp` to correctly copy `n_gpu_layers` to `cparams_mtp` and disable `offload_kqv` when `n_gpu_layers == 0`, preventing CUDA VRAM OOM on 8GB GPU.

---

## 4. First-Principles Non-linear Latency Model in TG Phase for $(K, p_{\text{min}}, n_{\text{min}})$

Under `-ot "exps=CPU"` for the weight-quantized `IQ4_NL` model, CPU MoE computation dominates the target verification latency $T_{\text{tgt}}(M)$ (where $M = K + 1$ is the verify batch size). 
Measured TG latencies under taskset P-core binding and NUMA interleave:
* $T_{\text{tgt}}(1) = 44.66$ ms (baseline)
* $T_{\text{tgt}}(2) = 79.15$ ms ($K=1$)
* $T_{\text{tgt}}(3) = 83.53$ ms ($K=2$, extremely small delta of +4.38 ms from $K=1$)
* $T_{\text{tgt}}(4) = 100.64$ ms ($K=3$, large delta of +17.11 ms from $K=2$)

### 4.1 Expected Cost per Token Model
The expected latency per generated token is modeled as:
$$\tau(K, p_{\text{min}}, n_{\text{min}}) = \frac{\mathbb{E}[\text{Time}]}{\mathbb{E}[A] + 1}$$
where $T_{\text{dft}} = 1.50$ ms. For fixed draft lengths $N$:
* $Time(1) = 1 \cdot 1.50 + 79.15 = 80.65$ ms.
* $Time(2) = 2 \cdot 1.50 + 83.53 = 86.53$ ms.
* $Time(3) = 3 \cdot 1.50 + 100.64 = 105.14$ ms.

Using typical acceptance probabilities $p_1 = 0.85$, $p_2 = 0.65$, $p_3 = 0.40$:
* **For $N=1$:** $\mathbb{E}[A] = 0.85 \implies \tau(1) = \frac{80.65}{1.85} = 43.59$ ms (1.024x speedup)
* **For $N=2$:** $\mathbb{E}[A] = 1.4025 \implies \tau(2) = \frac{86.53}{2.4025} = 36.02$ ms (1.24x speedup)
* **For $N=3$:** $\mathbb{E}[A] = 1.6235 \implies \tau(3) = \frac{105.14}{2.6235} = 40.08$ ms (1.114x speedup)

Thus, the global optimum is at $K=2$ ($n_{\text{max}} = 2$). The latency penalty of going to $K=3$ (+17.11 ms) outweighs the marginal gain of the 3rd token.

### 4.2 Early Stopping and Minimum Draft Length
We derive the marginal gain threshold for draft token probabilities $q_i$:
1. **Step 2 Threshold (Continue to $N=2$ vs Stop at $N=1$):**
   $$\tau(2) < \tau(1) \implies \frac{86.53}{1.85 + p_1 p_2} < 43.59 \implies p_2 > 0.16$$
2. **Step 1 Threshold (Continue to $N=1$ vs Stop at $N=0$):**
   $$\tau(1) < \tau(0) \implies \frac{80.65}{1 + p_1} < 44.66 \implies p_1 > 0.806$$

Since the engine only accepts a single $p_{\text{min}}$ parameter, we introduce $n_{\text{min}} = 2$ to resolve this mismatch. By clearing any draft of size $< 2$ to 0:
- If $q_1 < 0.16$, we abort (properly avoiding the net loss).
- If $q_1 \ge 0.16$ and $q_2 < 0.16$, we abort at step 2. The draft size is 1, which is cleared to 0 (avoiding the inefficient $N=1$ verification).
- If $q_1 \ge 0.16$ and $q_2 \ge 0.16$, we verify $N=2$ tokens (achieving the optimal $1.24x$ speedup).

Expected speedup under this optimized configuration ($K=2$, $p_{\text{min}} = 0.16$, $n_{\text{min}} = 2$):
$$\tau = \frac{0.8075 \times 86.53 + 0.1925 \times 44.66}{0.8075 \times 2.4025 + 0.1925 \times 1} = 36.80 \text{ ms/token (1.214x speedup)}$$
This is a **62% relative improvement in speedup** over the old parameters (1.132x).

Thus, the optimal server configuration is:
* **`--spec-draft-n-max 2`**
* **`--spec-draft-p-min 0.16`**
* **`--spec-draft-n-min 2`**
